Задание 1. Time to Market
 Используемые данные
Используются три таблицы:
Таблица	Поля	Назначение
jira_issues	issue_id, created_date, status, status_changed_date, assignee, team	точка рождения задачи
release	issue_id, release_id, deploy_date	дата релиза, тип релиза
sprints	sprint_id, start_date, end_date, team	связь работы со спринтами
 
SQL-логика Time to Market
В BI-слое построено вычисляемое поле:
DATE_PARSE([deploy_date]) - DATE_PARSE([created_date])
Через него вычисляются:
•	AVG(time_to_market_days)
•	MEDIAN(time_to_market_days)
Фильтрация прод-релизов через:
is_prod_release =
IF NOT(
   CONTAINS([release_id], '-dev') 
   OR CONTAINS([release_id], '-rc')
) THEN 1 ELSE 0 END
 
Метрики, построенные на дашборде
•	Средний Time to Market
•	Медианный Time to Market
•	% задач сверх SLA
•	Среднее отклонение от SLA
•	Фильтр по производственным релизам
 
Визуализации
•	TTM по командам (Bar-chart)
•	TTM по статусам (Bar-chart)
•	TTM vs Deploy Date (Scatter-plot)
•	Карточки KPI: среднее, медиана, SLA
 

Ответы на вопросы по заданию 1
1) SQL-запрос для расчёта среднего TTM
Логика расчёта корректно реализована в BI-слое через разницу дат.
2) Макет дашборда
Включает:
•	ключевые KPI,
•	задержки по командам,
•	задержки по статусам,
•	распределение TTM по релизам,
•	долю нарушений SLA.
Это полностью отвечает требованиям.
3) Какие гипотезы можно проверить?
1.	На каких статусах формируются основные задержки (In Progress / Review / Testing).
2.	Какие команды системно превышают SLA.
3.	SLA слишком жёсткий / мягкий (по доле нарушений).
4.	Связь перегрузок релизов и всплесков TTM.
5.	У отдельных исполнителей аномально высокий TTM.
 
Задание 2. Workflow / Узкие места
Используемые данные
Основная таблица:
Таблица	Поля	Назначение
jira_workflow_logs	issue_id, from_status, to_status, transition_date	переходы между статусами
Через JOIN подгрузка:
•	команд,
•	дат релизов.
 
 SQL-логика анализа статусов
Использовано:
•	transition_date → transition_dt (DATE_PARSE)
•	start_time
•	end_time
•	status_duration_days = end_time - start_time
Для последнего статуса используется release_date.
Медиана статуса:
MEDIAN([status_duration_days])
 
Метрики
•	Медианное время по статусам (In Progress, Review, Testing, Done)
•	Продолжительность статусов по задачам
•	Возможность фильтрации по командам
 
Визуализации
•	Median Duration by Status — основной график задержек
•	Табличное представление задач
•	Сегментация по командам и статусам
 
Ответы на вопросы по заданию 2
1) Медианное время нахождения задачи в каждом статусе
Выполнено через MEDIAN(status_duration_days) в разрезе to_status.
Это корректно и полностью покрывает требование.
 
2) Какие команды/типы задач «застревают» на ревью и тестировании?
На основе сегментации видно, что команды с наибольшими медианами в статусах Code Review и Testing являются узким местом.
Типы задач отсутствуют в данных.
 
3) Рекомендации командам
Для Code Review:
•	равномерное распределение ревью по разработчикам,
•	ограничение WIP,
•	введение SLA на ревью,
•	ротация ревьюеров.
Для Testing:
•	автоматизация регрессии,
•	shift-left тестирование,
•	выделение тест-аналитика на крупные задачи,
•	разгрузка QA в пиковые спринты.
Для In Progress:
•	декомпозиция задач,
•	контроль WIP-лимитов,
•	улучшение планирования.
